Implementation Notes — Phase 1

Diff Dad Review Loop · Review experience (diff-first guided walkthrough) · decisions not covered by the spec, for human review

Walkthrough model — derived, not stored

Context
The spec's Data Model puts walkthrough and reviewReady in the Zustand store (WalkthroughState).
Decision
Derive both via pure selectors over narrative + files instead of storing them. The store already mutates narrative incrementally (applyPlan/applyChapter/applyPartialNarrative), so a derived buildWalkthrough recomputes for free on every stream update and keeps the builder the single seam.
Alternative
Store walkthrough and recompute it inside all three streaming actions — rejected as a stale-data trap (miss one action → stale beats) that also duplicates the seam. The scout flagged this and recommended deriving.

review-store — safeStorage guard (pre-existing bug)

Context
The spec assumed store-driven tests need no infra, but review-store.ts read localStorage at module-eval time (store init) and in ~9 methods. localStorage is undefined in Vitest's node worker and bun test, so importing the store threw — the existing useLiveStream.test.ts was already red under both runners, and the new gate test could not run.
Decision
Added a capability-detected safeStorage wrapper (typeof localStorage !== 'undefined', no-ops otherwise) and rewrote every call site. Repairs the existing test; full web suite is now 15/15; npx vitest run (a spec validation command) is green.
Alternative
Add jsdom + a test.environment — rejected for Phase 1: heavier dependency, slower tests, and the spec explicitly prefers no DOM infra. safeStorage also fixes a real latent bug (the store crashed in any non-browser context), not just the tests.

Walkthrough model — chapterIndex threaded onto beats & resolve items

Context
The spec's Beat/ResolveItem types carry id/beatId but no numeric chapter index. The "Ask dad" action (POST /api/ai {action:'ask', chapterIndex}) requires a numeric 0-based chapterIndex and the server 400s without it.
Decision
Added chapterIndex: number to both Beat and ResolveItem. Real beats carry their 0-based index; the synthetic "Other" beat uses -1 (sentinel: no originating chapter, so "Ask dad" is disabled there).
Alternative
Re-derive the index by parsing beat.id (ch-N) in the resolve strip — rejected as brittle string-parsing; the explicit field is clearer and the other beat has no parseable index.

Walkthrough model — beats keyed ch-${index}, not themeId

Context
The spec's failure-mode table says "key beats by stable themeId" to avoid streaming reorder flicker.
Decision
Key beats by ch-${index}. The store replaces chapters at their index (applyChapter), so order is already stable by plan index, and ch-${index} matches the existing data-chid="ch-N" anchors that useScrollTracker walks — so rail scroll-tracking works with zero changes. themeId is optional on Chapter (absent on single-pass/older narratives), making it an unreliable key.
Alternative
Dual-key by themeId with an index fallback — rejected: adds a keying path and breaks the existing anchor convention for no gain, since index order is already stable.

Walkthrough model — orphan concerns default to info human decision

Context
Concerns whose file matches no chapter ("orphans") have no owning chapter risk to synthesize a severity from, but must stay visible — a dropped concern is a hidden review issue.
Decision
Collect orphans into a trailing "Other" beat (mirrors OrphanedInlineComments); each orphan resolve item defaults to info severity. Confirmed with the human, who chose info over warn to keep the rail quiet — treating an unplaced concern as low-confidence noise.
Alternative
warn — treat an orphan as possibly-misfiled-and-important and flag it louder. Considered and rejected by the human in favor of the noise-minimizing default.
