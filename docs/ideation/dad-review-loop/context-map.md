## Key Patterns

**`packages/web/src/components/Chapter.tsx`** — The resolution seam to replicate in `buildWalkthrough`. `findHunk(files, file, hunkIndex)` (lines 161-168) normalizes the path, finds the `DiffFile`, indexes `diffFile.hunks[hunkIndex]`, returns `null` if either misses — this null-drop behavior is exactly the "missing-hunk graceful drop" the builder needs. `Chapter` consumes a `Chapter` (theme) and renders `chapter.sections` (alternating `{type:'narrative'}` / `{type:'diff'}`), `chapter.whyMatters`, `chapter.callouts`, `chapter.reshow`, `chapter.risk` (`'low'|'medium'|'high'`). Note: `risk` here is 3-valued (low/medium/high) but the spec's `Beat.risk` is 4-valued (`'risk'|'warn'|'info'|'none'`) — the builder must map between these vocabularies. The streaming-shimmer pattern lives here too: `pendingChapterThemeIds.has(chapter.themeId)` gates a "Writing chapter prose…" indicator (lines 178-181, 300-326).

**`packages/web/src/state/types.ts`** — Input shapes the builder consumes (these are the web mirrors of the CLI types):

- `NarrativeResponse = { title; tldr; verdict: 'safe'|'caution'|'risky'; readingPlan; concerns: Concern[]; chapters: Chapter[]; missing? }`
- `Chapter = { title; summary; whyMatters; risk: 'low'|'medium'|'high'; sections: Section[]; callouts?; reshow?; themeId? }`
- `Section = { type:'narrative'; content } | { type:'diff'; file; startLine; endLine; hunkIndex }`
- `Concern = { question; file; line; category: ConcernCategory; why }` — **note: `Concern` has `category`, NOT `severity`.** The spec's `ResolveItem.severity: 'risk'|'warn'|'info'` does not exist on `Concern`; the builder must derive severity (likely from the owning beat's `risk`, since `concerns` carry no severity of their own).
- `DiffFile = { file; isNewFile; isDeleted; hunks: DiffHunk[]; mtime? }`; `DiffHunk = { header; oldStart; oldCount; newStart; newCount; lines }`.
- `themeId?` is optional on `Chapter` — the spec's failure-mode mitigation ("key beats by stable `themeId`") must tolerate its absence (older/single-pass narratives); fall back to index.

**`packages/web/src/components/StoryView.tsx`** — Hosts the TOC + reading surface in a `grid-cols-[220px_minmax(0,1fr)]` (line 244); `<ChapterTOC />` is the left rail, `<main>` maps `narrative.chapters` → `<Chapter>`. Calls `useScrollTracker()` at the top (line 215). **`OrphanedInlineComments` (lines 17-101) is the orphan-attach pattern the spec says to mirror**: it builds a `Set` of rendered hunk keys (`${normalizePath(file)}:${hunkIndex}`), finds inline comments whose hunks aren't in that set, and renders them in a trailing section so they're never invisible. The builder's "orphan concern → trailing 'Other' beat" should follow this shape. `StoryView` early-returns `null` if `!narrative` (line 219) — this is the gate the diff-first change must bypass.

**`packages/web/src/components/ChapterTOC.tsx`** — Evolves into `BeatRail`. Today: reads `narrative`, `activeChapterId`, `chapterStates`, `setActiveChapter`; renders one entry per chapter with a number/✓ badge, title, `hunkCount · risk · commentCount`; `jump(id)` calls `setActiveChapter(id)` then `document.querySelector('[data-chid="${id}"]').scrollIntoView(...)`. Returns `null` if `!narrative` (line 49). The rail must add: risk-flag glyph, "N to resolve" header, and read `toResolve`/beats from the walkthrough model rather than `narrative.chapters` directly.

**`packages/web/src/hooks/useScrollTracker.ts`** — Active-section tracking to reuse. Subscribes to `window` scroll; at `scrollY + 180`, walks `[data-chid="ch-N"]` elements bottom-up and calls `setActiveChapter`. Keyed entirely on `[data-chid]` attributes and `narrative.chapters.length`. To reuse for beats, the reading surface must emit matching `data-chid` anchors per beat (Chapter already emits `data-chid="ch-${index}"`).

**`packages/web/src/hooks/useNarrative.ts`** — The non-blocking pattern to mirror (the `watch` branch, lines 40-61): sets `setGenerating(false)`, populates the store via `useReviewStore.setState({ pr, files, comments, ... narrative: null })`, and `WatchView` renders immediately. The PR path (lines 62-82, the `data.generating && !data.narrative` branch) already stores `pr`/`files`/`comments`/`checkRuns`/`reviews` but calls `setGenerating(true)` — that flag plus `App.tsx`'s gate is what blocks. The spec's "add a `reviewReady` signal" plugs in here.

**`packages/web/src/App.tsx`** — The blocking gate is line 133: `if ((generating || !narrative) && files.length === 0) return <GeneratingScreen .../>`. **Critical: this already falls through to render once `files.length > 0`** (the comment at lines 130-132 says so), routing to `<ClassicView />` when `!narrative` (line 142-143). So the diff is _already_ non-blocking for the Files view — the Phase-1 work is to render the _new review view_ (diff + `BeatRail` + reading surface) on the `!narrative` branch instead of `ClassicView`, and to ensure `GeneratingScreen` only shows in the genuine pre-files window. `mode === 'watch'` short-circuits to `<WatchView/>` at line 126 (keep this untouched — manual test #4).

**`packages/web/src/hooks/useLiveStream.ts`** — SSE handlers already wired (the events the walkthrough must react to are emitted today):

- `plan-ready` → `applyPlan(data.plan)` (lines 211-220) — synthesizes a skeleton narrative with placeholder chapters and populates `pendingChapterThemeIds`.
- `chapter-ready` → `applyChapter(data.index, data.chapter, data.themeId)` (lines 222-230) — replaces chapter at index, clears its `themeId` from pending.
- `narrative.partial` → exported `handleNarrativePartialEvent` → `applyPartialNarrative` (lines 39-53) — **this is the unit-test entry point pattern**: a pure-ish exported function that drives the store, tested without DOM in `useLiveStream.test.ts`.
  Because the walkthrough is _derived_ from `narrative`+`files`, the simplest design is to recompute `buildWalkthrough` whenever those change (selector/memo), so no new SSE handler is strictly required — the spec's "ensure events update the walkthrough" is satisfied by deriving rather than storing-and-mutating. Confirm which approach the builder takes against the `WalkthroughState` shape in the spec (which stores `walkthrough` in the store).

**`packages/web/src/hooks/useComments.ts`** — "Send to agent" pattern. `postComment(body, opts)` already routes to `POST /api/agent-comments` when `mode === 'watch'` (lines 34-54) with `{ path, line, side: 'RIGHT', startLine, startSide, body }`. **Caveat for review (PR) mode**: in PR mode `postComment` posts to `/api/comments` (GitHub), NOT the agent. The resolve strip's "Send to agent" must call `/api/agent-comments` _directly_ (not via `postComment`) if it needs the agent loop regardless of mode — or the spec's assumption that "Send to agent" works in the review path needs the agent store to exist in PR mode. **Flag for builder: verify `ctx.store` (agent-comment store) is initialized on the `dad review` path, not just `dad watch`.**

**`packages/cli/src/server.ts` `/api/ai` (lines 296-446)** — "Ask dad" endpoint. The `ask` action (lines 411-417) requires: a numeric `chapterIndex` (400 if missing, line 378), `ctx.narrative` present (**503 "narrative still generating"** if not, line 382), and a valid chapter at that index (400 otherwise). Request: `{ action:'ask', chapterIndex:number, question:string }`. Response: `{ text:string }` (or `{ error }`). **Two implications for the resolve strip**: (1) a `Beat` is keyed by `id`/`themeId`, not a numeric chapter index — the builder must carry the originating 0-based chapter index onto the beat (or resolve item) so "Ask dad" can pass it; (2) "Ask dad" will 503 until the narrative finishes generating — the strip must handle this (the spec's error-handling table covers "ask fails → inline error").

**`packages/cli/src/server.ts` `/api/agent-comments` POST (lines 465-495)** — Requires `body` (string) and `path` (string) + `line` (number); optional `side`/`startLine`/`startSide`/`hunkContext`/`chapterTitle`. Returns the created `AgentComment` and broadcasts an `agent-comment` SSE. The resolve strip's "Send to agent" needs the beat's `file`+`line` — which come from `ResolveItem.file`/`line` (themselves from the originating `Concern.file`/`Concern.line`, 1-based new side).

**`packages/cli/src/narrative/prompt.ts`** — Output-streamlining target. Caps live as module constants: `MAX_CHAPTERS = 7`, `MAX_LINES_PER_FILE = 800`, `MAX_TOTAL_DIFF_LINES = 12000`, `MAX_THEMES = 7` (lines 46-49). Summary-length guidance is prose inside `SYSTEM_PROMPT`/`PLANNER_SYSTEM_PROMPT`/`WRITER_SYSTEM_PROMPT` (e.g. "summary — exactly 1 sentence", "narrative sections 1-3 sentences", the "Brevity" section lines 239-241). **Important correction to the spec: `NARRATIVE_MAX_TOKENS=16384` is NOT in `prompt.ts` — it lives in `packages/cli/src/narrative/engine.ts:30`** (used only by the single-pass `buildNarrativePrompt` path at engine.ts:366). The two-pass path uses `PLANNER_MAX_TOKENS = 6000` (`planner.ts:27`) and `WRITER_MAX_TOKENS = 4000` (`writer.ts:23`). To "lower the guide's effective token budget" the builder must touch the constant relevant to whichever path is active — for streaming PRs that's the planner+writer budgets, not `NARRATIVE_MAX_TOKENS`.

## Conventions

- **Naming**: Components PascalCase `.tsx` in `components/`; hooks `useX.ts` in `hooks/`; pure libs in `lib/` (e.g. `paths.ts`); tests in co-located `__tests__/` as `X.test.ts(x)`. Store is `review-store.ts` with `useReviewStore` + a `create<ReviewState>()` Zustand store.
- **Imports**: Relative paths throughout (`../state/review-store`, `./Hunk`); no barrel/index re-exports; type-only imports use `import type`. No path aliases.
- **Error handling**: SSE handlers wrap `JSON.parse` in `try/catch` and silently ignore malformed payloads (the established "ignore malformed event" idiom). Store normalizes streamed narratives at the boundary (`sanitizeNarrative`, review-store.ts:213) because partials arrive with missing array fields — `.map`/`.filter` on `undefined` would crash the React tree. `buildWalkthrough` should assume inputs _may_ be mid-stream (empty `sections`, missing `themeId`) and never throw.
- **Types**: `type` aliases (not `interface`) for data shapes in `state/types.ts`; discriminated unions for `Section`/`LiveEvent`. The web `state/types.ts` and CLI `narrative/types.ts` are parallel hand-maintained mirrors (CLI uses `NarrativeChapter`/`NarrativeSection`; web uses `Chapter`/`Section`) — keep them aligned but they are separate files. CLI also has a separate `plan-types.ts` (planner/writer import `Plan`/`PlanTheme` from there, not `types.ts`).
- **Testing**: `packages/web` uses Vitest ^3, `test: "vitest run"`, **no `test` block in `vite.config.ts`** → default node environment, **no jsdom / Testing Library**. The sole existing web test (`useLiveStream.test.ts`) tests an _exported pure function_ and drives the Zustand store via `setState`/`getState` with `beforeEach` reset — **this is the exact pattern both new tests must follow**. CLI uses `bun test` against `packages/cli/src/__tests__/`. Root `typecheck` covers both packages (`package.json:15`).

## Risks

- **Spec inaccuracy — `NARRATIVE_MAX_TOKENS` location.** The spec says to lower it "from `NARRATIVE_MAX_TOKENS=16384`" in `prompt.ts`, but the constant lives in `engine.ts:30` and only governs the _single-pass_ fallback. The streaming two-pass path (the one the spec wants to keep "preferred") is bounded by `PLANNER_MAX_TOKENS`/`WRITER_MAX_TOKENS` in `planner.ts`/`writer.ts`. Streamlining the wrong constant will not move the streaming latency the spec targets.
- **`Concern` has no `severity`.** The spec's `ResolveItem.severity: 'risk'|'warn'|'info'` has no source field on `Concern` (which carries `category` + a 1-based `line`). The builder must synthesize severity — most likely from the owning beat's `risk` (mapped `high→risk`, `medium→warn`, `low→info`). Make this mapping explicit and test it; otherwise severity is undefined behavior.
- **Risk-vocabulary mismatch.** `Chapter.risk` is `'low'|'medium'|'high'`; `Beat.risk` is `'risk'|'warn'|'info'|'none'`; `TriageFlag.severity`/`ResolveItem.severity` are `'risk'|'warn'|'info'`. Three overlapping vocabularies — the builder is the single place to reconcile them; a wrong mapping silently mis-flags beats in the rail.
- **"Ask dad" keying + 503 window.** The `/api/ai` `ask` action needs a numeric `chapterIndex` and returns 503 until `ctx.narrative` exists. Beats are keyed by `id`/`themeId`. The builder must thread the originating 0-based chapter index onto each beat/resolve item, and the strip must tolerate the 503 while the guide streams (the diff-first promise means the strip can appear before the narrative is done).
- **"Send to agent" in PR/review mode.** `useComments.postComment` only routes to `/api/agent-comments` in `watch` mode; in `review` mode it posts to GitHub. The resolve strip must call `/api/agent-comments` directly, and the agent-comment store (`ctx.store`) must be confirmed initialized on the `dad review` server path — the spec assumes "the endpoint exists" but the _store_ may only be wired for watch. Verify before relying on it.
- **Derived-vs-stored walkthrough.** The spec's `WalkthroughState` stores `walkthrough` in the Zustand store, but the SSE events already mutate `narrative` incrementally. If the walkthrough is _stored_ it must be recomputed on every `applyPlan`/`applyChapter`/`applyPartialNarrative`; if it's _derived_ (selector/memo over `narrative`+`files`) it updates for free. Picking the stored model without wiring recompute into all three store actions will make beats stale on stream. Deriving is lower-risk and keeps `buildWalkthrough` the single seam.
- **No DOM tests = `BeatRail`/`ResolveStrip`/`App` render paths are unverified by CI.** Per the spec this is intentional (pure-function/store tests only), but regressions in the rail, the diff-first gate wiring in `App.tsx`, and the resolve-strip UI won't be caught automatically — they rest entirely on the manual dev-server checklist. Acceptable for Phase 1, but worth stating: the gate test asserts store state, not that `App` actually renders the review view instead of `GeneratingScreen`.
