The problem

01Diff Dad turns a GitHub PR into a narrated, browsable diff and syncs comments bidirectionally with GitHub. But during local development — while an AI agent (typically Claude Code in a sibling tmux pane) iterates on a branch — there is no way to leave a review comment that feeds back to the agent. Every comment dad can post goes to GitHub.

02The result is a broken inner loop: the developer reviews the agent's work in dad, then has to manually retype or copy-paste feedback into the agent's pane, and has no visibility into whether the agent saw a comment or what it did about it. Inspired by hunk.dev's watch mode, we want a local review-while-the-agent-edits loop where comments are delivered to the agent over MCP and the loop visibly closes (each comment shows delivered → addressed, with the agent's reply inline).

Goals ×5
01 Run `dad watch [base]` inside a git repo to review the local working-tree diff (against the merge-base with the default branch by default) with the existing narrative UI — no PR or GitHub token required.
02 Leave comments in watch mode that reach a connected agent via an MCP server mounted on dad's existing Hono server (3 tools: list_review_comments, reply_to_comment, resolve_comment).
03 Make the loop visible: every agent comment moves open → delivered (auto, on MCP fetch) → addressed, the agent can reply inline, and replies/status update live in the UI via the existing SSE channel.
04 Provide a manual fallback (clipboard button in the UI, `dad comments` markdown to stdout) so any agent that can't speak MCP can still consume the comments.
05 Persist agent comments to the cache dir so they survive dad restarts and agent context resets.

MVP — must ship ×6
  • `dad watch [base]` local mode: shell out to `git diff`, parse with existing diff-parser, synthesize PRMetadata, generate the narrative, serve the existing UI.— The irreducible entry point — without a local diff source there is nothing to comment on.
  • Agent comment store (packages/cli/src/agent-comments/) with open→delivered→addressed lifecycle, replies, and write-through persistence to the cache dir.— Single source of truth shared by the UI and MCP; the durable core of the loop.
  • MCP server at /mcp (new @modelcontextprotocol/sdk dep) exposing all 3 tools, registered before the static catch-all. GATED BY A SPIKE: first prove the SDK's Streamable-HTTP transport bridges to Hono's Request/Response (via c.req.raw) and survives `claude mcp add --transport http`; the rest of the MCP work proceeds only after the transport handshake is confirmed.— The primary delivery channel; locked decision. No MCP SDK exists in the repo today and mounting an HTTP-transport MCP server inside Hono is unproven — the spike de-risks the single highest-uncertainty item before the tools are built.
  • Compose UI for agent comments + inline agent replies/status, reusing CommentThread/Comment. Requires an ADDITIVE `source`/`status` field on PRComment (coexisting with — not replacing — the existing id<0 draft heuristic), plus widening PRComment.id/inReplyToId and groupThreads from number to string|number (AgentComment.id is a string) with an audit of numeric-id call sites.— Makes the loop visible — the core DX win. Reuse is real but not free: the thread renderer's numeric-id assumption and id-sign provenance are load-bearing and must be widened, not bypassed.
  • Manual fallback: 'Copy for agent' button (navigator.clipboard) and `dad comments` stdout, both rendering open comments as markdown.— Cheap, portable escape hatch for non-MCP agents; the locked secondary channel.
  • Make ServerContext.github nullable and guard every GitHub-dependent path: /api/checks, /api/comments GET+POST, /api/review, and the recap path (startRecapGeneration + /api/recap), which crash the Recap tab in watch mode. The SSE poller body (server.ts:372-539, which calls ctx.github ~5x/tick) is REPLACED wholesale with a debounced working-tree watcher in watch mode, not merely flag-disabled. Keep the server alive while unresolved agent comments exist (drop the 'MCP client connected' condition — stateless HTTP tools have no liveness source of truth).— Required plumbing — watch mode has no GitHub client, the recap/poller paths dereference it unconditionally, and the agent needs /mcp alive after the browser closes.
Out of scope — said no on purpose
  • Auto-injecting comments into the agent via tmux send-keys. — The agent pulls via MCP or the user pastes; no keystroke injection.
  • Rich thread management (assignees, labels, multiple custom states). — B-lite lifecycle only — three states is deliberately the ceiling.
  • Any GitHub destination in watch mode (no GitHub-vs-Agent toggle, no Promote-to-GitHub). — `dad watch` is a pre-PR tool — there is no PR to route or promote to; watch-mode comments are purely agent-directed.
Future — someday, maybe
  • PR-mode agent routing: a GitHub-vs-Agent destination toggle in the composer when reviewing an already-open PR (deferred — only useful once you want the loop on real PRs, which the pre-PR watch workflow does not need).
  • 'Promote to GitHub' on a resolved agent thread, reusing the existing postComment path (deferred with PR-mode routing — requires an actual PR to exist).
  • 'Code changed → likely addressed' heuristic: the watch loop flags a delivered comment whose underlying lines changed even if the agent never called resolve_comment.
  • Per-agent targeting when more than one agent/pane is connected to the MCP server.
  • Comment templates / canned review snippets for common feedback.
