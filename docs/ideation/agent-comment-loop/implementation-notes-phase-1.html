Implementation Notes — Phase 1

MCP transport — stateful session map, not stateless

Context
The spec's step-1 spike said to prove the MCP SDK's Streamable-HTTP transport bridges to Hono, with a Bun.serve second-port fallback if it could not. It did not specify stateful vs. stateless.
Decision
Used WebStandardStreamableHTTPServerTransport (Fetch-native, with a literal Hono example in the SDK docs) mounted at /mcp in stateful mode with a sessionId → transport map. The SDK forbids reusing a stateless transport (webStandardStreamableHttp.js:137-139) and a fresh transport rejects any non-initialize request (:590), so stateless cannot serve the multi-request initialize → tools/call flow. The Bun.serve fallback was not needed.
Alternative
Stateless + fresh-transport-per-request (rejected: breaks tools/call), or a standalone Bun.serve listener on a second port (rejected: unnecessary — the in-Hono mount works, proven by an in-memory initialize/tools-call handshake test).

Test runner — server-importing tests require bun test, not vitest

Context
The spec's success-criteria check commands used npx vitest run. But any test that imports server.ts transitively loads hono/bun, whose ssg.js references the Bun global at module load — undefined under Node-vitest, so those suites fail to even load.
Decision
Run the suite with bun test src/__tests__/ (the root package.json test script and CLAUDE.md already use this). Pure-logic tests (store, mcp tools/transport, local diff) pass under both runners; server/route/SSE tests need Bun. Validation uses bun test.
Alternative
Add a vitest config that shims the Bun global or aliases hono/bun — rejected as out-of-scope churn when a Bun-native runner already exists.

Pre-existing failing test — GET /api/narrative config assertion

Context
server.test.ts "returns the full narrative when ready" asserts data.config matches { theme: 'auto', … }, but the handler returns readConfig() — the developer's real config (here theme: "light"). The test is not isolated from the on-disk config.
Decision
Left as-is — it predates this work and is unrelated to the agent loop (the /api/narrative config block was not modified). Flagged so the 274-pass / 1-fail result isn't mistaken for a regression.
Alternative
Mock readConfig in that test — deferred as out-of-scope test-quality cleanup.

Web UI — self-contained agent drawer instead of CommentThread reuse + id-widening

Context
The spec called for reusing CommentThread/Comment and widening PRComment.id/inReplyToId/groupThreads to string | number. The scout flagged that this blast radius also reaches Hunk.tsx, StoryView.tsx, useLiveStream.ts, and review-store.ts (all numeric-id-keyed), plus the ReplyBox prop — a wide, risky type change across the GitHub comment pipeline.
Decision
Built agent comments as a distinct type (AgentComment, string ids + status lifecycle) with its own store field (agentComments), its own SSE handler (agent-comment), a useAgentComments hook, and a self-contained AgentComments drawer (compose + live status/replies + "Copy for agent"). The numeric-id GitHub pipeline is untouched — zero widening, zero churn in Hunk/StoryView/review-store.
Alternative
The spec's inline reuse (click a diff line → compose → render in-thread) is nicer UX but forces agent string-ids through the numeric-id store and Hunk renderer. Deferred as a follow-up; the drawer delivers the full visible loop (compose → delivered → addressed + replies) at a fraction of the risk.

Accepted review findings (non-blocking MVP tradeoffs)

Context
The cycle-1 review (PASS, 0 critical/high) raised three quality findings deemed acceptable for the MVP.
Decision
(1) Polling vs debounce — the watch loop uses a 2000ms setInterval gated by watchRegenerating + a content-hash no-op guard, rather than the spec's ~500ms trailing debounce. Simple and robust for a single-user local tool; a true fs-watch debounce is a follow-up. (2) Stale-on-clean — if the user reverts all edits, buildLocalReview throws CleanTreeError, which the watcher swallows, leaving the last narrative served. Accepted; clearing the view on a newly-clean tree is a follow-up. (3) Store key churn — the no-base store key uses the merge-base SHA, which is stable across working-tree edits (the property that matters in-session) but moves when the default branch advances, orphaning prior comments. Accepted for short-lived pre-PR sessions.
Alternative
Each has a clear follow-up (fs-watch debounce, clean-tree broadcast, branch-name keying); none blocks the MVP loop.
